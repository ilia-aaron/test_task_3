## Архитектура

Проект построен по методологии `Feature-Sliced Design (FSD)`. Простая масштабируемость, слабая связность и предсказуемая структура папок.

- `app/` — инициализация приложения, глобальные стили и провайдеры, темизация.
- `pages/` — сборка страниц. Страница `ProjectsPage` компонует виджеты фильтров и таблицы.
- `features/` — бизнес-логика и интерактивные блоки:
  - `filters` — сложная панель фильтрации с контекстом состояния (`FiltersContext`). Включает фильтры по департаменту, статусу, приоритету и сложный фильтр по менеджерам (`ManagerFilter`).
  - `projects-table` — таблица с интеграцией автоподгрузки данных.
- `entities/` — бизнес-сущности (Project, Manager), их типы, мапперы и API сервисы.
- `shared/` — переиспользуемый код: UI компоненты (`GpnTable`), утилиты, константы.

Почему выбрано имеено FSD для небольшого приложения? Потому что любое небольшое приложение в будущем может стать большим, и FSD позволяет легко масштабировать приложение.

## Реализация

### 1. Фильтр менеджеров (`ManagerFilter`)

Фильтрация менеджеров — самая сложная часть задачи, реализованная с учетом высокой нагрузки и UX.

- Если пользователь нажимает "Выбрать всех" и снимает галочки с пары менеджеров, мы не отправляем 248 параметров `managerId`.
- Вместо этого алгоритм вычисляет, выбрано ли больше половины всех менеджеров. Если да, запрос инвертируется и отправляются только снятые галочки оператором `_ne` (например, `?managerId_ne=50&managerId_ne=67`).
- Если выбраны все (без исключений) — параметр `managerId` вообще очищается, чтобы запросить все проекты.
- Это избавляет сервер от парсинга гигантских URL и ускоряет ответ.
- Список из 250 менеджеров рендерится через `react-virtuoso`. Реализован пагинируемый запрос к `/managers`. При скролле вниз данные подгружаются в привычном формате бесконечной ленты.
- В логике кнопки "Выбрать всех" учитывается есть ли поисковой фильтр. Если есть, то "Выбрать всех" выбирает всех менеджеров, которые соответствуют поисковому фильтру. Если нет, то "Выбрать всех" выбирает всех менеджеров. Далее исключение из этого списка ведется в exluded Set().
- Для удобства было решено добавить инверсирование списка менеджеров, с помощью таба "Выбрать / Исключить". Т е если пользователь хочет исключить из выборки несколько менеджеров, то ему будет удобнее выбрать таб "исключить" и выбрать тех которых не должно быть в выборке во избежание путаницы.

### 2. Таблица AG Grid (Server-Side Row Model)

Для работы с 10 000 записей используется `ag-grid-enterprise` с `Server-Side Row Model`:

- Создан `createServerSideDatasource`, который конвертирует параметры грида (модель запроса AG Grid) в параметры API (пагинация `_page`/`_limit`, сортировка `_sort`/`_order`).
- Фильтры из FSD-фичи `Filters` прокидываются на уровень таблицы и инициируют пересоздание `DataSource`, заставляя таблицу обновить данные.
- Встроенный в AG Grid `loadingOverlayComponent` по умолчанию ограничивается размерами одной видимой строки.
- Поэтому `Loader` от Consta вынесен поверх таблицы через `position: absolute / inset: 0`, а события перехватываются внутри `DataSource` (через методы `success` и `fail`), чтобы показывать индикатор загрузки на всю высоту блока.
- Loader показывается только при первоначальной загрузке, чтобы при пагинации не блокировать таблицу целиком.
- Бюджет и затраты форматируются с разделением тысяч и символом рубля (например, `1 200 500 ₽`). Заменить отображение можно в одном месте, сделав изменения в утилите форматирования. Сейчас она использует Intl.NumberFormat - международные утвержденные форматы отображения чисел.
- Прогресс выводится в формате `XX%` + прогресс бар с разными цветами по уровням, опять же настраивающаяся история.
- Цвета статусов, приоритетов и прогресса брал из css переменных чтобы было проще поддерживать консистентность с остальной частью приложения. Если в будущем понадобится изменить цвета, то это можно будет сделать в одном месте.

Note:

- Почему выбрал именно Server-Side Row Model вместо своих велосипедов?
- Я приверженец того что если есть что то готовое что работает и не требует доработок, то лучше использовать его для ускорения и упрощения разработки. При чем я полностью даю себе отчет какие могут быть подводные камни и как их избежать. Именно сейчас я выбрал Server-Side Row Model, для ускорения разработки. Но при разработке в enterprise среде я бы предпочел написать свой велосипед, так как он бы позволил мне иметь полный контроль над всем процессом и избежать подводных камней.

### 3. Сетевой слой

- Написан собственный сервис-обертка над нативным `fetch` (Для того что бы показать возможность Dependency Invertion).
- Автоматически сериализует объекты с массивами в формат QueryString стандарта json-server (дублирование ключей: `?status=Активный&status=Завершён`).
- Реализован с заделом под переиспользование инфраструктуры (ретраи, интерцепторы авторизации).
- Данные с сервера проходят через `projectDtoToProjectMapper` для нормализации в бизнес-модели.
- Параметры поиска наоборот трансформируются через `projectSearchParamsToDto` для подгонки под синтаксис json-server (`managerId` -> `managerId_ne`).

Note:
Это база без которой не начинаю ни 1 приложение. Позволяет быстро и безболезненно переезжать между разными API. И иметь одну точку истины - свою модель и не зависеть от изменений со стороны бэкенда API.

## Что можно улучшить

- Добавить тесты
- Добавить сценарии обработки ошибок
- Добавить кэширование где нужно
- Я бы не использовал напрямую компоненты из@consta/uikit, нужно вынести это в независимый слой UI, фасадом, а внутри если нужно использовать @consta/uikit. Это позволит в будущем безболезненно переезжать на другой UI фреймворк без переписывания всего приложения.
- Добавить Tailwind CSS для стилизации. Позволяет быстрее верстать и точечней управлять стилями. Опять же это не отменяет необходимости использовать UI компоненты, но дает больше гибкости.
- Провести аудит зависимостей, проверить на уязвимости. (11 vulnerabilities (3 moderate, 8 high) - нужно исправить)
